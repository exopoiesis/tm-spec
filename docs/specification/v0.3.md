# TM-Spec v0.3 — Draft

> Date: 2026-06-01. Status: [DRAFT].  
> Schema: [`../../schemas/0.3.json`](../../schemas/0.3.json).  
> v0.3 is an additive extension of v0.2: all v0.2 documents remain valid
> after changing `spec: tm-spec/0.2` to `spec: tm-spec/0.3`, and a v0.2
> document continues to validate against `0.2.json` unchanged. The
> reference validator selects the schema from each document's `spec:`
> field.

## 1. Header

Every TM-Spec v0.3 document MUST include:

```yaml
spec: tm-spec/0.3
kind: <Kind>
id: tm.<slug>.<YYYY-MM-DD>
schema_url: https://exopoiesis.github.io/tm-spec/0.3.json
```

## 2. Core sections inherited from v0.2

v0.3 keeps every v0.2 section and kind unchanged:

- `structure`, `defects`, `magnetic`, `environment`, `reactions`,
  `calculation`, `workflow`, `results`, `sanity`, `provenance`
- kinds: `NEBCalculation`, `USCalculation`, `MetaDynCalculation`,

  `RelaxCalculation`, and the compositional fragments `Structure`,

section-by-section narrative. The machine contract is the v0.3 JSON
Schema. v0.3 adds two fields and one optional section.

## 3. New field: `endpoint.geometry_origin`

Each NEB endpoint (under `workflow.endpoints.<label>`) gains an optional
`geometry_origin` field recording how that endpoint geometry was obtained:

```yaml
workflow:
  kind: NEB
  endpoints:
    A:
      ref: artifacts/endA.extxyz
      E_eV: -128055.5404
      fmax: 0.027
      geometry_origin: dft_relaxed
    B:
      ref: artifacts/endB.extxyz
      E_eV: -128055.5402
      fmax: 0.026
      geometry_origin: dft_relaxed
```

Allowed values:

- `dft_relaxed` — geometry relaxed at the DFT level used for the run.
- `dft_static` — geometry from a DFT single-point (no relaxation).
- `mlip_relaxed` — geometry relaxed with a machine-learning interatomic
  potential.
- `experimental` — taken from an experimental structure.
- `as_built` — constructed by hand / from a prototype without relaxation.
- `unknown` — origin not recorded.

**Why it matters.** Energy comparisons across endpoints are valid only
when both endpoints share a `dft_relaxed` origin. An `mlip_relaxed`
geometry can carry tens of eV of unrelaxed-lattice error even when local
bond lengths look physical, which silently invalidates an energy
difference. The corresponding pre-flight check is
`G09_geometry_origin` in [`../gate-registry.md`](../gate-registry.md).

The field name is deliberately **not** `provenance`: the top-level
`provenance` block records *record authorship* (date, author, parents,
compute), in the AiiDA/QCSchema sense, whereas `geometry_origin` is a
per-endpoint statement about how a *geometry* was produced.

### 3.1 Standards alignment

`geometry_origin` is a human shortcut in the spirit of `magnetic.state`:
there is no single scalar field for it in the established standards, so it
exports rather than imports. The `(method × relaxation)` value maps onto
emmet `run_type × task_type` (`dft_relaxed` ~ `"<XC> Structure
Optimization"`, `dft_static` ~ `"<XC> Static"`) and onto NOMAD
`electronic_structure_method` (an `mlip_relaxed` origin ~ `method=ML`);
QCSchema expresses the "final optimized geometry" structurally via
`OptimizationResult.final_molecule` rather than as a label. The
`preflight` block has **original** semantics with no standard equivalent
(it is the forward-looking twin of the post-hoc `sanity` array); only its
`engine{name, version, routine}` sub-object is aligned, onto QCSchema
`provenance{creator, version, routine}`. See
[`../standards-alignment.md`](../standards-alignment.md) §12 for the full
mapping tables.

### 3.2 Structure-level `geometry_origin` (single-calc docs)

`geometry_origin` also lives, optionally, on `structure` (added to
`$defs.structure` in 0.3, additive -- no version bump). A NEB doc has one
geometry **per endpoint**, so its origin belongs on each
`workflow.endpoints.<label>.geometry_origin`. A single-structure doc
(`SinglePointCalculation` / `RelaxCalculation`, e.g. a NOMAD import) has no
`endpoint` block, so the natural home for the origin is
`structure.geometry_origin`. Both use the same enum and both surface through
`G09_geometry_origin`. Rule of thumb: **endpoint-level for NEB,
structure-level for single-structure docs.** The NOMAD importer now sets
`structure.geometry_origin` directly (and still emits the G09 sanity gate that
the prodromos planner reads).

## 4. New optional section: `preflight`

`preflight` is an optional top-level block carrying a **predictive
pre-flight assessment** produced by an external engine (the reference
implementation is **Prodromos**) BEFORE the expensive calculation is run.
It is the forward-looking counterpart of the post-hoc top-level `sanity`
array: both speak the same gate-ID vocabulary defined in
[`../gate-registry.md`](../gate-registry.md). `sanity` records what was
checked after the run; `preflight` records what an engine predicted
before committing compute, so the two can be compared on the same IDs.

`preflight` is OPTIONAL and is not part of any kind's required sections.

```yaml
preflight:
  engine:
    name: prodromos
    version: "0.1.0"
  verdict: GO                       # GO / NO-GO / INVESTIGATE / REVIEW
  confidence: high                  # low | medium | high
  plan_graph_version: "2026-06-01"
  gates:
    - id: G03_endpoints_distinct    # same vocabulary as sanity[]
      verdict: ASYMMETRIC
      confidence: high
      p_success: 0.92               # calibrated, lower credible bound
      reasons: ["endpoints differ by a single proton hop"]
      next_actions: ["proceed: well-posed minimum-energy path"]
    - id: G09_geometry_origin
      verdict: ENDPOINT_VALID
      confidence: high
      p_success: 0.97
    - id: G12_endpoint_single_sheet
      verdict: GO
      confidence: medium
      p_success: 0.80
  plan:
    next_action: "Launch 7-image NEB from the two dft_relaxed endpoints."
    next_action_cost_usd: 40.0
    strategies:                     # optional scored strategy tree
      - label: direct_neb_7img
        method: "CI-NEB, 7 images"
        expected_cost_usd: 40.0
        p_success: 0.88
        cvar_usd: 85.0              # conditional value-at-risk (tail), USD
        utility: 0.81
        paper_grade_reachable: true
```

### 4.1 `preflight.gates[]`

Each gate uses the same `id` pattern as `sanity` (`^G\d{2}_[A-Za-z0-9_]+$`)
and the same registry. Required per gate: `id`, `verdict`. Optional:
`confidence`, `p_success` (calibrated probability, lower credible bound,
in `[0, 1]`), `reasons`, `next_actions`.

### 4.2 `preflight.plan`

Two complementary shapes:

- **route mode** — a single recommended `next_action` (+ optional
  `next_action_cost_usd`).
- **tree mode** — a `strategies[]` array of scored candidate next
  workflows. Each leaf may carry `expected_cost_usd`, `p_success`,
  `cvar_usd` (tail risk), `utility`, `paper_grade_reachable`, and
  `proposed_workflow_ref` (a pointer to a draft TM-Spec document for that
  strategy).

## 5. Gate registry

The gate-ID vocabulary shared by `sanity[]` and `preflight.gates[]` is
the canonical [`../gate-registry.md`](../gate-registry.md). One ID means
the same check in both places; the only difference is *when* it is
evaluated. A predictive engine MUST emit IDs from that registry (or
propose additions via a spec PR).

## 6. Compatibility

- A v0.1 document MUST validate against `0.1.json`.
- A v0.2 document MUST validate against `0.2.json`.
- A v0.3 document MUST validate against `0.3.json`.
- The reference validator selects the schema from the document's `spec:`
  field; `SUPPORTED_VERSIONS = ("0.1", "0.2", "0.3")`.
- v0.2 documents are NOT auto-rewritten. They stay at `tm-spec/0.2` and
  remain valid; only documents that use `endpoint.geometry_origin` or the
  `preflight` block need `spec: tm-spec/0.3`.

## 7. Implementation notes

Reference implementation version `0.3.0` keeps the v0.2 module surface
(`tm_spec.validator`, `extract`, `lint`, `sanity_fill`,
`exporters.nomad`, `importers.nomad`, `cli`). The spec text is
intentionally concise; exact required/optional fields are defined by the
JSON Schema and guarded by tests.
